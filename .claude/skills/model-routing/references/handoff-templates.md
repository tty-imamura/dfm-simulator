# ハンドオフ文書テンプレート

上位モデルの成果物を下位モデルに渡すための「実行指示書」の書き方。

## 原則

下位モデルは**曖昧さの解決が苦手**なだけで、明確な指示の実行は得意。したがってハンドオフ文書の役割は、実行中に発生しうる全ての判断を**事前に潰しておく**こと。文書を書き終えたら「下位モデルがこの文書だけを読んで、質問ゼロで完走できるか?」と自問する。1つでも「そこは常識で判断してくれるだろう」と思った箇所があれば、それが品質崩壊の起点になる。

良いハンドオフ文書の判定基準:
- **判断語が残っていない**: 「適切に」「いい感じに」「必要に応じて」が出てきたら書き直し。具体値・具体例・分岐条件に置換する。
- **インターフェースが契約になっている**: 関数シグネチャ、JSONスキーマ、ファイルパスが確定している。下位モデルが命名やデータ形状を発明する余地がない。
- **正例と負例がある**: 期待する出力の実例と、「これはダメ」の実例。few-shotは下位モデルに対して最も費用対効果の高い補償策。
- **受け入れ条件が検証可能**: 「動くこと」ではなく「`npm test` が全パス」「このcurlがこのJSONを返す」。

## テンプレート1: 設計 → 実装ハンドオフ

```markdown
# 実装指示書: [コンポーネント名]

## 目的(1段落)
[何を作るか、なぜ作るか。実装者が迷ったときの判断基準になる]

## 成果物
- ファイル: [正確なパスと役割の一覧]
- 変更してよいファイル / 変更禁止のファイル

## インターフェース契約
[関数シグネチャ、型定義、JSONスキーマ、APIエンドポイント仕様を確定形で。
 「〜のような」は禁止。コピペで使える形で書く]

## 実装手順
1. [順序が重要な場合は番号付きで。各ステップの完了条件も書く]

## エッジケースと対応(表)
| 入力/状況 | 期待動作 |
|-----------|---------|
[空入力、上限超過、ネットワーク失敗、並行アクセス等、想定しうるものを網羅]

## 使用技術の制約
- 使ってよいライブラリ(バージョン込み)/ 禁止ライブラリ
- コーディング規約(既存コードのスタイル例を1つ貼る)

## 正例・負例
[小さくてよいので、期待するコードの実例と、避けるべき書き方の実例]

## 受け入れ条件(全て検証可能な形で)
- [ ] [コマンドとその期待結果]

## 禁止事項
[スコープ外の変更、勝手なリファクタ、依存追加など、やりがちな逸脱を明記]
```

## テンプレート2: 実装 → 実行時プロンプト(アプリに焼き込むプロンプト)

実行時APIで下位モデル(主にHaiku)が使うプロンプトは、開発時に上位モデルが作成し、検証してからアプリに埋め込む。

```markdown
# 実行時プロンプト仕様: [処理名]

## 呼び出しコンテキスト
- モデル: [model string]
- 頻度: [1ユーザー操作あたり何回/月間想定]
- レイテンシ要件: [ms]
- max_tokens: [値と根拠]

## システムプロンプト(確定版)
[役割、入出力形式、制約を含む完成品。プレースホルダは {var} 形式で明示]

## few-shot例(3〜5個)
[本番データに近い入出力ペア。エッジケースを最低1つ含める]

## 出力スキーマ
[JSONスキーマ。パース失敗時のリトライ方針も書く]

## 検証結果
[開発時にこのプロンプトをテスト入力N件で検証した結果。
 合格率、失敗パターン、フォールバック条件(Sonnetに逃がす条件)]
```

## テンプレート3: レビュー指示書(上位モデルによる品質ゲート)

```markdown
# レビュー指示: [対象フェーズ]

## 入力
- ハンドオフ文書: [パス]
- 成果物: [パス]

## 検査項目(優先順)
1. 受け入れ条件との差分を全て列挙せよ(合格項目も明記)
2. インターフェース契約からの逸脱を列挙せよ
3. エッジケース表の各行が実際に処理されているか確認せよ
4. [フェーズ固有の観点]

## 出力形式
- 判定: PASS / FAIL
- FAIL項目ごとに: 場所、問題、**ハンドオフ文書への追記案**(修正指示は文書改訂の形で出す)
```

## 書き方のコツ

- **文書は上位モデルの1回分のコストで、下位モデルの何十回分の品質を買う投資**。ここをケチらない。
- 手戻りが起きたら、修正指示を口頭で出すのではなく**ハンドオフ文書を改訂**する。文書が改善されれば、同種の次のタスクにも効く。
- 長すぎる文書は下位モデルの注意を散らす。「網羅的だが冗長でない」を目指し、背景説明よりも具体値・表・実例に紙幅を使う。
