# 小型モデルの性能を最大化する技法

下位モデル(Haiku等)を上位モデルの品質に近づけるための技法カタログ。実行時APIの設計時と、開発フェーズで下位モデルに作業を委譲する時に参照する。

## 選定の考え方

コストと効果のバランスで、まず上から順に検討する:

| 技法 | 追加コスト | 効く場面 |
|------|-----------|---------|
| 1. タスク分割 | ほぼゼロ | ほぼ常に。最初に検討 |
| 2. few-shot | 入力トークン増(キャッシュで相殺可) | 形式模倣、分類、抽出 |
| 3. 構造化出力 | ほぼゼロ | JSON等の機械可読出力 |
| 4. プロンプトキャッシュ | 削減側 | 長い固定プロンプトの反復呼び出し |
| 5. 検証+リトライ | 失敗時のみ増 | 出力が機械検証可能な場合 |
| 6. Best-of-N | N倍 | 品質分散が大きく、選別が容易な場合 |
| 7. 上位モデルフォールバック | 失敗時のみ上位料金 | 難易度が入力によりばらつく場合 |
| 8. SFT/蒸留 | 初期投資大 | 超高頻度の単一タスク |

## 各技法の要点

### 1. タスク分割
小型モデルは「1呼び出しで複数の判断」に弱い。判断1つ=呼び出し1つに分割すると、各呼び出しの難易度がルーブリック上1段下がる。例: 「レビューを読んで感情を判定し、カテゴリ分類し、返信文を書く」→ 3呼び出しに分割。Haiku3回はSonnet1回より安いことが多い(要現行料金確認)。

### 2. few-shot
下位モデルへの最強の補償策。3〜5例、本番分布に近いもの、エッジケースを最低1つ。例は上位モデルに生成・選別させ、開発時に固定する。「例の質」が支配的なので、上位モデルによる例のレビューを挟む。

### 3. 構造化出力
自由記述を許すと下位モデルは形式が揺れる。JSONスキーマを明示し、「スキーマ以外を出力するな」と指示。パース失敗時は失敗理由を添えて1回だけリトライ、それでも失敗ならフォールバック。

### 4. プロンプトキャッシュ
システムプロンプト+few-shot部分をキャッシュ対象にすると、反復呼び出しの入力コストが大幅に下がる。「few-shotを厚くすると高い」問題の多くはこれで解消するため、キャッシュ前提で例を厚めに入れる設計が成立する。仕様と料金は docs.claude.com で確認。

### 5. 検証+リトライ
出力を機械検証(スキーマ、テスト、値域チェック)し、失敗時のみ失敗内容をフィードバックして再実行。成功率90%のタスクなら期待コストは1.1回分で済み、実効品質は大きく上がる。

### 6. Best-of-N
同一入力でN回生成し、機械的スコアまたは安価な判定呼び出しで最良を選ぶ。生成の分散が大きいタスク(創作、命名、要約)で有効。N=3程度から。選別コスト<品質向上価値のときだけ使う。

### 7. 上位モデルフォールバック(段構え)
まずHaikuで処理し、(a)機械検証失敗、(b)モデル自身の確信度低下(「判定不能」を出力させる設計)、(c)入力が事前定義の難条件に該当、のいずれかでSonnetへ。難入力が全体の1〜2割なら、全件Sonnetよりずっと安く、品質はほぼ同等になる。

### 8. SFT/蒸留
上位モデルの入出力ペアで下位モデルをファインチューニング。単一タスクを月数十万回以上呼ぶ規模で初めて投資が回収できる。まず1〜7で足りないことを確認してから。

## 組み合わせの定石

**実行時APIの標準構成**(まずこれを検討):
```
プロンプトキャッシュ(固定部) + few-shot 3〜5例 + 構造化出力
+ 機械検証 → 失敗時リトライ1回 → なお失敗でSonnetフォールバック
```
この構成は追加実装が軽く、Haiku単体比で品質が大きく向上し、コストはほぼHaiku水準に収まる。

**開発時に下位モデルへ委譲する場合の標準構成**:
```
密度の高いハンドオフ文書(handoff-templates.md 準拠)
+ 受け入れ条件の機械検証 + 上位モデルによるフェーズ末レビュー
```
