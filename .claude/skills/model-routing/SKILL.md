# Model Routing — 過不足のないモデル選定と低性能モデルの性能最大化

## 基本原則

1. **品質バーが先、コストが後。** まず成果物の品質基準(受け入れ条件)を定義し、それを満たせる**最も安いモデル**を各フェーズに割り当てる。品質を落としてコストを下げるのは失敗。時間がかかるのは許容される。
2. **仕様の密度が能力差を補償する。** 高性能モデルと低性能モデルの差は「曖昧さの解決能力」に現れ、「明確に指定された作業の実行能力」の差は小さい。したがって、上位モデルが曖昧さを全て潰した**ハンドオフ文書**を書けば、下位モデルは上位モデルに近い品質で実行できる。これがこのスキルの核心。
3. **レビューは生成より安い。** 下位モデルの出力を上位モデルが検査するのは、上位モデルが最初から生成するより大幅に安い。フェーズ境界に品質ゲートを置く。
4. **知能を消費で終わらせず、資産化する。** 上位モデルの呼び出し結果(設計書・プロンプトテンプレート・few-shot例)はファイルとして保存し、下位モデルが何度でも再利用できる形にする。

## モデル階層(2026年時点)

| Tier | モデル | model string | 得意領域 | 相対コスト |
|------|--------|--------------|----------|-----------|
| 最上位 | Claude Fable 5 | `claude-fable-5` | 最難関の設計判断、新規性の高いアーキテクチャ、曖昧で矛盾を含む要求の整理 | 最高 |
| 上位 | Claude Opus 4.8 | `claude-opus-4-8` | 設計、レビュー、複雑なデバッグ、ハンドオフ文書の作成 | 高 |
| 中位 | Claude Sonnet 4.6 | `claude-sonnet-4-6` | 仕様が明確な実装、リファクタ、テスト作成、中難度の分析 | 中 |
| 下位 | Claude Haiku 4.5 | `claude-haiku-4-5-20251001` | 実行時API(分類・抽出・整形・簡易応答)、大量バッチ処理 | 低 |

**注意**: モデルのラインナップと料金は変わる。コスト見積りを出す前に、必ず https://docs.claude.com で現行のモデル一覧と料金を確認すること。料金を暗記で書かない。

## ワークフロー

### ① タスク分解
タスクをフェーズに分解する。典型的な分解軸:
- **設計** — 要件整理、アーキテクチャ、データモデル、UI/UX方針
- **実装** — コード生成、ドキュメント作成、アセット組み込み
- **実行時** — 完成物が運用中に行うAPI呼び出し(これは開発時と別枠で毎回課金され続けるため、最もコスト感度が高い)
- **レビュー/QA** — フェーズ境界での検査
- **外部AI生成** — 画像・動画・音声・3D等、Claude以外の生成AIに委譲する部分

### ② フェーズ難易度判定
各フェーズを以下のルーブリックで採点し、必要Tierを決める:

| 判定質問 | Yesなら |
|----------|---------|
| 要求に曖昧さ・矛盾・未決定事項が残っているか | 上位以上(Fable/Opus) |
| 前例のない構造・アルゴリズムの発明が必要か | 上位以上 |
| 複数ファイル・複数制約を横断する整合性が必要か | 中位以上(Sonnet) |
| 仕様書どおりに書き下すだけの作業か | 中位で十分 |
| 入出力形式が固定で、判断が分類・抽出・整形に閉じるか | 下位で十分(Haiku) |
| 1回の失敗コストが高い(不可逆・公開される・課金される)か | 1 Tier引き上げ |
| 実行回数が非常に多い(実行時API、バッチ)か | 1 Tier引き下げを検討し、仕様密度で補償 |

**迷ったら**: そのフェーズだけ両Tierで小さく試作し、上位モデルに blind 比較させて決める。試作コストは誤選定コストより安い。

### ③ モデル割当とルーティング表の作成
割当結果は必ず**ルーティング表**として明示する(後述の成果物フォーマット)。「なんとなくSonnet」は禁止。各行に割当理由と、下位モデルを使う場合の補償策(ハンドオフ文書・few-shot・検証ゲート)を書く。

### ④ ハンドオフ文書の作成(このスキルの中核)
上位フェーズの成果物は、**下位モデルがそれだけを読んで迷わず実行できる文書**でなければならない。「設計書」ではなく「実行指示書」を書く意識を持つ。テンプレートと書き方の原則は `references/handoff-templates.md` を読むこと。**下位モデルに仕事を渡すフェーズがあるなら、このファイルを読まずにハンドオフ文書を書かない。**

### ⑤ 品質ゲート
フェーズ境界ごとに検査を置く:
- **機械的検査を最優先**: テスト、lint、スキーマ検証、ビルド成功。モデル費用ゼロ。
- **モデル検査**: 機械検査で拾えない品質(設計意図との整合、エッジケース漏れ)は、そのフェーズより1 Tier上のモデルにレビューさせる。レビュー指示には「ハンドオフ文書の受け入れ条件との差分を列挙せよ」と具体的に書く。
- 不合格なら、修正指示を**ハンドオフ文書への追記**として書き戻し、同じ下位モデルに再実行させる(文書が改善資産として蓄積される)。

### ⑥ 実行時APIの設計(該当する場合)
成果物がアプリで、運用中にClaude APIを呼ぶ場合:
- デフォルトはHaiku。Haikuで品質が出ない処理だけSonnetに逃がす**フォールバック段構え**を設計する。
- 小型モデルの性能を引き上げる技法(few-shot、構造化出力、タスク分割、Best-of-N、プロンプトキャッシュ等)は `references/small-model-techniques.md` を読んで適用する。
- 実行時プロンプトは開発時に上位モデルが作成・検証し、アプリにはテンプレートとして焼き込む(原則4の資産化)。

### ⑦ 外部AIへの委譲
画像・動画・音声・3D・楽曲など、Claudeが直接生成できない/外部の専用AIが優れている成果物は、**外部AI向け指示書(ブリーフ)**を作成して委譲する。指示書の構造とツール別の書き分けは `references/external-ai-briefs.md` を読むこと。指示書なしに「いい感じの画像を作ってください」とユーザーに丸投げしない — 指示書の品質が生成物の品質を決める。

### ⑧ コスト見積りと報告
ルーティング表に、フェーズごとの概算トークン量と概算コストを付す。実行時APIがある場合は「1リクエストあたり」と「想定月間」の両方を出す。料金は必ず docs.claude.com で確認した現行値を使い、確認日を書く。

## 成果物フォーマット

計画立案を求められたら、必ずこの構成で出す:

```markdown
# [タスク名] モデルルーティング計画

## 品質基準(受け入れ条件)
- [成果物が満たすべき条件を検証可能な形で列挙]

## ルーティング表
| フェーズ | 担当モデル/外部AI | 割当理由 | 補償策 | 品質ゲート | 概算コスト |
|---------|------------------|---------|--------|-----------|-----------|

## ハンドオフ文書
[フェーズ間の受け渡し文書。references/handoff-templates.md 準拠]

## 外部AI指示書(該当時)
[references/external-ai-briefs.md 準拠]

## コスト見積り
- 開発時: [内訳]
- 実行時: [1req / 月間](該当時)
- 料金確認日: YYYY-MM-DD(docs.claude.com)
```

計画だけでなく実行まで求められた場合は、この計画に従い、各フェーズで指定Tierの視点・制約を守って順に実行する(自分が単一モデルであっても、ハンドオフ文書と品質ゲートの規律は同様に適用する — 規律自体が品質を作るため)。

## アンチパターン

- **全部上位モデル**: 品質は出るがコスト目標に反する。仕様が固まった実装をOpusにやらせるのは浪費。
- **全部下位モデル**: 曖昧さ解決を下位に任せると、手戻りコストが節約分を上回る。
- **ハンドオフ文書の手抜き**: 「詳細は適宜判断」と書いた瞬間、下位モデルの品質は崩れる。曖昧さを残すなら、そのフェーズは上位モデルの担当。
- **料金の暗記**: 古い料金でコスト計算して意思決定を誤る。必ず現行値を確認。
- **レビューなしの直列パイプ**: 下位モデルの誤りが後段に伝播し、最終段で高くつく。
